### **Файл: AGENTS.md**

### **Техническое задание (Refactoring & Architecture): Стабилизация ядра, исправление API и синхронизации**

**Вводная для агента:**
Текущая кодовая база имеет серьезные архитектурные изъяны в микросервисном взаимодействии, обработке статики и управлении базой данных. Наблюдаются регулярные ошибки `500 Internal Server Error`, сбросы соединений (`ERR_CONNECTION_RESET`), ошибки `404 Not Found` для загруженных медиа и полная рассинхронизация данных между React-фронтендом и Telegram-ботом. 
Ваша задача — провести глубокий рефакторинг бэкенда и инфраструктуры, опираясь на принципы "Единого источника истины" (Single Source of Truth) и асинхронной обработки тяжелых задач.

---

#### **Блок 1. Исправление инфраструктуры медиафайлов (Ошибки 404)**
**Проблема:** Изображения, загружаемые через админ-панель (в `static/uploads/`), не отдаются сервером Nginx, возвращая 404. Это происходит из-за изоляции файловых систем контейнеров.
**Задача:**
1. **Настройка Docker Volumes:** В файле `docker-compose.yml` (и `.prod.yml`) необходимо создать общий (shared) `volume` для статических файлов.
2. Пробросить этот volume в контейнер `backend` (куда сохраняются файлы при загрузке через API) и в контейнер `nginx` (откуда они должны раздаваться).
3. **Конфигурация Nginx:** Проверить файл конфигурации Nginx (`nginx.conf` / `default.conf`). Добавить или исправить блок `location`, чтобы он корректно маршрутизировал запросы вида `GET /static/uploads/...` в примонтированную директорию.

#### **Блок 2. Стабилизация Backend API (Устранение ошибок 500)**
**Проблема:** CRUD-операции (особенно `PUT` запросы на обновление цен тарифов `api/v1/admin/plans/{id}`) вызывают падение сервера и возвращают 500 ошибку.
**Задача:**
1. **Глобальный перехват исключений:** Внедрить на уровне FastAPI (или используемого фреймворка) глобальный Exception Handler. Приложение никогда не должно "падать молча". Ошибка должна логироваться в консоль (с полным `Traceback`), а фронтенду должен возвращаться читаемый JSON с описанием ошибки.
2. **Управление сессиями БД:** Проверить логику работы с SQLAlchemy (или другой ORM). Убедиться, что сессии базы данных корректно открываются и закрываются (`yield` в зависимостях / `finally` блоки).
3. **Безопасные транзакции:** Обернуть логику обновления (`UPDATE`) в блоках эндпоинтов в конструкцию `try...except...finally`. Если происходит сбой валидации или конфликт в БД, транзакция должна откатываться (`db.rollback()`), предотвращая повреждение базы и зависание коннектов.

#### **Блок 3. Асинхронная архитектура рассылок (Устранение Connection Reset)**
**Проблема:** Рассылка из веб-панели вызывает `net::ERR_CONNECTION_RESET`. Веб-панель пытается отправить 1000+ сообщений в рамках одного синхронного HTTP-запроса, что приводит к таймауту веб-сервера.
**Задача:**
1. **Разделение инициализации и выполнения:** Эндпоинт `/api/v1/admin/broadcast-image` должен быть переписан. Теперь его задача — **только принять данные** от веб-панели (текст, картинку, фильтры) и мгновенно вернуть HTTP-статус `202 Accepted` ("Задача принята в обработку").
2. **Делегирование Воркеру:** Само выполнение рассылки (цикл отправки сообщений в Telegram API) должно передаваться в фоновую очередь (использовать `Celery`, если он настроен, либо `BackgroundTasks` из FastAPI). Бэкенд не должен ждать завершения рассылки, чтобы ответить фронтенду.

#### **Блок 4. Единый источник истины (Решение проблем синхронизации)**
**Проблема:** Бот показывает старые кнопки, старые цены или не применяет картинки, хотя в админ-панели они изменены. Бот и веб-панель работают рассинхронизировано.
**Задача:**
1. **Удаление хардкода:** Провести аудит кода Telegram-бота. Полностью исключить использование "зашитых" в код текстов меню, названий кнопок, цен и конфигураций.
2. **Динамическая генерация:** Бот должен формировать клавиатуры (Reply/Inline) и текстовые сообщения **исключительно** на основе данных, получаемых из базы данных (или быстрого кэша Redis).
3. **Инвалидация кэша:** Если бот использует локальное кэширование для ускорения работы, необходимо внедрить механизм инвалидации. Любой `PUT/POST/DELETE` запрос из веб-панели, меняющий настройки (тексты, тарифы, кнопки), должен очищать соответствующий кэш бота, заставляя его при следующем запросе пользователя выгрузить свежие данные из БД.
4. **Универсальная привязка медиа:** Логика бота должна быть изменена: перед отправкой любого сообщения (по любой кнопке меню), бот обязан проверить в БД наличие привязанного к этой кнопке/сообщению `media_id` или URL изображения. Если изображение есть — отправлять `send_photo` с подписью (caption), если нет — обычное текстовое сообщение.
